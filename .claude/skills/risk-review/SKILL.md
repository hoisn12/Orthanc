# Risk Review

변경 사항에 대해 아래 리스크 항목을 점검한다.

## 컨텍스트 수집

- `git diff`로 변경된 코드 확인
- 변경이 외부 입력, API, 데이터 처리에 관련되는지 파악

## 보안

- [ ] 인젝션 취약점 (SQL, Command, XSS) 없는가
- [ ] 민감 정보(API 키, 비밀번호)가 하드코딩되지 않았는가
- [ ] 인증/인가 로직에 우회 가능 경로가 없는가
- [ ] 사용자 입력이 적절히 검증/이스케이프되는가

## 성능

- [ ] N+1 쿼리, 불필요한 루프, 대량 메모리 할당 없는가
- [ ] 비동기 처리가 필요한 곳에 동기 호출을 사용하지 않았는가

## 호환성

- [ ] 기존 API 계약(인터페이스, 반환값)이 깨지지 않았는가
- [ ] 하위 호환성이 필요한 경우 유지되는가

## 데이터

- [ ] 데이터 손실 위험이 없는가
- [ ] 마이그레이션이 필요한 경우 롤백 가능한가

## 출력

발견된 리스크를 심각도(High/Medium/Low)와 함께 보고한다.
High 리스크가 있으면 반드시 사용자에게 알리고 승인을 받는다.
리스크가 없으면 "리스크 없음"으로 보고하고 워크플로우를 완료한다.
